# Feature delivery handbook

CloudForge 使用“产品需求文档 → 技术设计与开发手册 → Issues → 实现与验证”的顺序交付新能力。模板不是填写任务；每一节都必须减少开发者需要自行猜测的内容。

## 1. 创建文档

1. 复制 [`PRD 模板`](../templates/prd-template.md) 到 `docs/product/<feature>-prd.md`。
2. 完成产品边界、领域语言、用户故事、业务规则和验收标准。
3. 复制 [`技术设计模板`](../templates/technical-design-template.md) 到 `docs/architecture/<feature>-technical-design.md`。
4. 使用 PRD 的 `FR-*` 和 `AC-*` 编号建立需求、实现与测试的可追踪关系。

PRD 解释“为什么做、用户得到什么、怎样算完成”；技术设计解释“数据归谁、接口怎样工作、故障如何处理、怎样证明实现正确”。不要在两个文档中复制同一段实现细节。

PRD 必须使用产品经理和业务人员能够直接理解的自然语言。领域术语可以保留，但接口路径、请求参数、状态码、表结构、字段类型、缓存、消息、事务、框架、部署和开发命令只能出现在技术设计中。

## 2. 评审门槛

进入开发前必须满足：

- 领域词汇与 `CONTEXT.md` 一致。
- 每个数据对象只有一个事实源。
- 用户接口、内部接口和第三方 Open API 的身份语义不混用。
- 每个租户请求明确身份来源、Tenant 来源、权限执行点和数据库隔离方式。
- 所有状态都有合法迁移、停用行为和历史保留规则。
- PRD 的每项功能需求都有验收标准。
- 技术设计的每项风险都有测试或运行时控制。
- 未决问题已经关闭，或明确标记为不阻塞的后续项。

## 3. 记录决策

产品决策记录在 PRD 的“依赖、风险与决策”，技术取舍记录在技术设计的“设计概述”或对应设计章节。新决策取代旧决策时直接更新对应文档并说明影响；不要创建独立 ADR，也不能让相互矛盾的产品或技术文档同时保持有效。

## 4. 拆分开发工作

批准后的 PRD 仍按仓库约定发布为 GitHub Issue。技术设计拆成可独立验证的子 Issue，顺序通常为：

1. 领域模型与数据库迁移。
2. 模块接口和持久化 Adapter。
3. 用户应用接口与内部接口。
4. Session、授权和租户隔离。
5. 导入、事件、审计与可观测性。
6. 端到端测试和运行手册。

每个 Issue 必须引用 PRD 的 `FR-*`/`AC-*` 和技术设计的 `TC-*`，并写明阻塞依赖与完成条件。

## 5. 实现与交付

- 按模块接口测试行为，不以私有方法和表结构作为主要测试接口。
- 先提交向后兼容的数据库迁移，再部署读取方和写入方。
- 运行 `make format`、`make check` 和 `git diff --check`。
- 按技术设计执行代表性端到端场景、故障路径和回滚演练。
- 实现与文档发生偏差时，先更新并重新确认决策，不能让代码悄悄成为另一套设计。
